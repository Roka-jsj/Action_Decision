# 클로드 스킬 & 하네스 카탈로그

이 프로젝트에서 추출한 재사용 자산의 전체 목록이다. 두 종류로 나뉜다:

- **클로드 스킬** (7종, [`.claude/skills/`](../.claude/skills/)) — Claude Code에서 슬래시 커맨드로 호출하는 절차 지식. 도메인이 달라도 적용된다.
- **하네스** (8종, `sim/` · `experiments_master.csv` · 운영 패턴) — 이 캠페인에서 실사고를 겪고 만든 실행 인프라. 스크립트는 그대로, 패턴은 이식해서 쓴다.

모든 항목에 "유래"를 적었다 — 전부 실제 사건에서 나왔고, 발명된 모범사례가 아니다.

## 1부. 클로드 스킬 7종

설치: 디렉토리를 `~/.claude/skills/`에 복사하면 모든 프로젝트에서 자동 인식된다. 호출: 대화에서 `/스킬이름` 입력 또는 해당 상황이 오면 Claude가 자동 참조.

### 1. `seed-sigma-first` — 분산 먼저 측정
- **언제**: 새 대회/캠페인 시작, 베이스라인 확정 직후
- **내용**: 시드 설정 코드 검증(작동하는지부터) → 동일시드 재실행 노이즈 ε 측정 → 시드간 σ 측정 → 현재 최고점이 몇 번째 시도의 max인지 계산(승자저주 체크). 목표까지의 갭이 4σ를 넘으면 그 축은 운영상 종료로 판정
- **유래**: 최고 멤버가 +2.7σ 행운 추첨이었음을 마지막 날 발견, 재현 시도에 GPU 슬롯 5개 소모. 시드 버그(torch.manual_seed가 set_seed를 덮음)도 종반에야 발견

### 2. `ml-transfer-audit` — 오프라인→실전 전이 감사
- **언제**: 새 아이디어의 제출 가치 판정
- **내용**: 개입을 기전 4클래스로 분류(강건화·지표정합 / 입력분포 정합 / 적합 강화 / 결정규칙) → 실험 원장에서 같은 계열의 과거 판정 대조 → 클래스별 전이계수로 기대 실전Δ 계산 → SUBMIT / HOLD / DEAD 판정
- **유래**: 홀드아웃 +0.005급 아이디어들이 리더보드에서 반복 음전. 결정규칙 재조정은 12전 12패 실측

### 3. `submission-guards` — 제출물 조립 안전가드
- **언제**: 제출 파이프라인 구축, 제출 오류 후 재발방지
- **내용**: 6종 가드의 bash 골격 — 자기파괴 방지(DST≠SRC) / 캐시 신선도 / 메타 assert / byte-diff / 용량(SI 바이트 기준, MiB 혼동 금지) / 실제 zip을 풀어 진입점을 끝까지 실행하는 런타임 캐너리
- **유래**: 개발용 import 잔존으로 서버 ModuleNotFoundError → 제출 슬롯 2개 소각. du의 879M(MiB)와 사이트의 920.9MB(SI) 혼동 소동

### 4. `train-fleet-watch` — GPU 학습 감시·함대 운영
- **언제**: 수 시간~밤샘 학습 발사 직후
- **내용**: PID 기반 4종 감시(완주/사망/정지/드라이버 붕괴), zip 쓰기중 레이스 방지(크기 안정 + testzip), 이중발사 감지(/proc/PID/environ 덤프), 마감 임박 시 자동 큐 중단 → 수동 원샷 전환, "상관≠인과 — 대조실험 먼저" 원칙
- **유래**: 커맨드라인 pgrep이 env변수를 못 봐 낸 오경보, 동일 시드 이중발사로 GPU 반분, 자동 큐가 학습 중인 GPU에 오발사, BadZipFile 레이스, gradient checkpointing 누명 사건(완벽한 상관이 대조실험 2발로 무죄)

### 5. `readout-protocol` — 판독 표준 프로토콜
- **언제**: 점수/평가 결과가 나올 때마다, 특히 예상 밖 판독
- **내용**: 평시(원장 기록 → 사전등록 분기 집행 → 다관점 판정 → 계획 델타만 보고) + 위기 5단계(무결성 검사 → 소형 라벨 프로브로 버그/품질 분리 → 단일변수 분리실험 → 가설별 사전확률·업데이트 규칙 등록 → 분기 실행). 이례 신호 감지 체크리스트(동률·연속 단조·오프라인/실전 역전) 포함
- **유래**: 서로 다른 두 모델이 0.00006차 동률로 폭락한 사건 — 이 프로토콜로 6시간 만에 원인(분포 진실) 확정

### 6. `quant-jitter-tickets` — SR 재양자화 재추첨
- **언제**: 대회 종반 GPU 병목 + 제출 슬롯 잉여, int8 배포 모델의 마지막 점수 회수
- **내용**: fp16 원본을 확률적 반올림(비편향)으로 재양자화 → 가중치 25%가 ±1LSB 플립 → 기대성능 동일한 새 티켓 (구현 코드 포함). 운용 규칙: k=1이 최적, 전 양자화 멤버 복합, 좋은 티켓의 행운 멤버는 고정하고 나머지만 재추첨
- **유래**: 이 프로젝트의 신규 발견. 9발 실측 σ≈0.0002, 최종 +0.0005 회수, 마지막 제출(~50% 베팅) 적중

### 7. `endgame-slots` — 마감일 슬롯 경제학·최종선택
- **언제**: 마감 24시간 전 계획, 마지막 슬롯 배분
- **내용**: 티켓별 P(목표)·P(경신)·기대폭 분리 계산, 버림서열 사전 확정, 시간 역산 하드컷, 체크박스 불변식(즉시 선체크 → "항상 현재 전역최대" 유지, 갱신은 풀 자릿수+제출ID+파일명 3중 대조, 새로고침 검증)
- **유래**: 점수 소수 3자리까지 같은 티켓이 5개 쌓여 오독이 실제 리스크가 된 마지막 아침

## 2부. 하네스 8종

### H1. 실험 원장 — `experiments_master.csv`
단일 CSV에 전 판독·판정·사고를 append (태그: LB실측/DIAG/VERDICT/PLAN/CONSTRAINT). 세 가지 역할: ①단일 진실원천 ②에이전트가 원장을 대조해 폐기된 접근의 재제안을 차단 ③사후 회고의 원자료. 이 캠페인에서 240행 축적.

### H2. 조립 파이프라인 — `sim/assemble_*.sh`
멤버 교체·재양자화 → run_meta 검증 → zip → 용량검사 → 런타임 캐너리를 원커맨드로 수행. 가드 6종 내장(스킬 3 참조). 인자화: `SRC_PKG`(스캐폴드 선택), `M1SRC`(임의 멤버를 지터 소스로) — 마감 밤 "행운 멤버 주변 재추첨"을 10분 작업으로 만든 설계.

### H3. 재양자화 도구 — `sim/quantize_member.py` · `quantize_member_jitter.py`
group-64 per-row int8 양자화(RTN)와 그 SR 지터판(`--qseed`, `--amp`). 대형 2D 텐서만 int8, 나머지 fp16 유지, 복원오차 리포트 내장. 1.32GB 멤버 2개를 1GB 제한 안에 넣은 도구이자 지터포켓 발견의 도구.

### H4. 런타임 캐너리 패턴
제출 zip 실물을 풀어 소형 목데이터로 진입점을 끝까지 실행하고 출력 행수·필수 마커·라벨 유효성을 assert. 서버에서만 터지는 오류(import 누락·경로·로더)를 잡는 유일한 지점. 잡지 못하는 것(시간초과·점수 품질)도 문서화되어 있어 과신을 방지.

### H5. 라벨 프로브 하네스 — 버그/품질 분리기
훈련 데이터 꼬리 N행(500/5k)으로 미니 테스트셋을 만들어 제출 패키지를 통째로 실행·채점. 용도: ①위기 시 기계적 버그와 품질 문제의 분리 ②raw/SWA 같은 변형 쌍의 paired bootstrap 신뢰구간 게이트. 주의사항 문서화: in-dist 점수는 히든과 순위가 역전될 수 있어 선별용으로는 금지.

### H6. GPU 발사·감시 — `sim/ops/launch_*.sh` + 모니터 패턴
- `launch_post_restart.sh`: CUDA 프로브 → 기존 학습 잔존 검사 → 양 GPU 발사 → 90초 생존확인
- `launch_hash6.sh`: 마감 시각 역산으로 가능 에폭 계산, 미확보 시 발사 취소(사전등록 룰 내장)
- 모니터 패턴(스킬 4에 스크립트 골격): PID 상태 + 로그 mtime + 산출물 출현 + CUDA 생존을 4분 주기 폴링, 이벤트만 통지

### H7. 멀티에이전트 판정 하네스
판독·위기·전략 수립마다 반복한 구조: 독립 관점 2~4개(법의학/재계획/반증사냥)를 병렬 실행 → 구조화 스키마(SUBMIT/HOLD/DEAD + 기대Δ + 근거)로 수렴 → 이견은 실측근거 우선으로 재정. 별도 모델 계열(codex CLI)의 독립 판정을 상호 비판시켜 교차검증 — 실제로 서로 다른 오류(한쪽의 임계값 오캘리브, 다른쪽의 EV 과장)가 교차에서 검출됐다. 사전등록 문서(실물: `docs/archive/r73_readout_prereg.md`)와 결합해 운영.

### H8. 자산 위생 절차
- 평가 전용/폐기 패키지는 격리 디렉토리로 이동(오업로드 방지)
- 디스크 정리는 명시적 삭제/보존 목록으로 (fp16 원본·미제출 티켓·원장은 절대 보존 — fp16 원본 보존이 지터포켓 축의 성립 조건이었다)
- 공개 push 전 보안 스윕: 추적 파일 + 전체 히스토리에서 키/개인정보/테스트셋 실물 검색, 독립 에이전트 이중 감사
